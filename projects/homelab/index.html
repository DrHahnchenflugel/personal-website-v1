Self-Hosted Infrastructure

A personally operated server stack for running production services without relying on managed platforms.

What this is

This is a self-hosted environment I use to run services: internal tools, personal apps, and infrastructure that needs to stay online and recover cleanly when something breaks.

It is not a demo setup. It exists because I need control over deployment, data, and failure modes.

Why I built it

I wanted an environment where I can:

  • Run services without external dependencies.
  • Understand exactly how traffic reaches each app.
  • Control upgrades, downtime, and recovery.
  • Fix problems directly instead of working around them.
Design constraint: All services run on hardware I control. If I can't deploy it, back it up, and restore it myself, it's not an acceptable dependency.

What runs on it

  • Internal web apps.
  • PostgreSQL databases with local backups.
  • Media and automation services isolated from core systems.
  • Reverse proxy and tunnel-based ingress.

Services are segmented. Not everything can see everything else, and nothing is exposed without intent.

Technical approach

Infrastructure

  • Proxmox host with VMs and containers.
  • Linux-based guests with minimal base installs.
  • Snapshots and backups treated as first-class requirements.

Application layer

  • Docker Compose for service definition.
  • Explicit networking between containers.

What this is not

  • Not a “cloud replacement” experiment.
  • Not optimized for infinite scale.
  • Not dependent on fragile automation.

The goal is reliability and understanding, not novelty.

Next steps

  • Stricter role separation for sensitive services.
  • More aggressive backup verification.
  • Documented recovery procedures for every critical service.
